Version 0.9 Release Notes

General Breaking Changes

Warning

Version 0.9 of Paya introduces changes that may break rigging code written in 0.8. These changes were necessary to clarify workflows, unify method interfaces and ringfence unpatched PyMEL scripts. Please read through these release notes carefully to understand where to perform refactoring if needed.

Non-Automatic Patching

Paya no longer automatically patches PyMEL as soon as paya.runtime is loaded. Patching must instead be started and stopped using paya.runtime.start() and paya.runtime.stop(), respectively, or, better yet, managed via new context-manager functionality on the paya.runtime object itself. The older behaviour is still available via a configuration flag. See Importing and Patching for details.

Background

This change was made to better ringfence external scripts that rely on unpatched PyMEL.

Persistent Native Units

As of version 0.9, all Paya methods will automatically switch Maya to native units before running, and switch it back to the original user setting when they’re done. The older behaviour is still available via a configuration flag. See Working with Native Units for more details.

Background

Paya wants you to rig with Maya configured for centimeters and radians. Before version 0.9 this was not sufficiently signalled to newcomers, leading to unexpected results when different units were used.

Consistent Sampler Method Behaviour

As of version 0.9, sampler methods across shapes and plugs will always default to a static (value) return unless plug=True is explicitly requested or another argument is passed-in as a plug. See Static vs Dynamic Sampling for more details.

Background

Before version 0.9, sampler methods on shape classes defaulted to static returns (i.e. values), while their counterparts on namesake plug classes defaulted to dynamic returns (i.e. live outputs). This made it less intuitive to predict default behaviour:

shape = r.PyNode('curve1').getShape()
print(shape.pointAtParam(0.5))
# Result: [-18.055226981265214, 0.0, 3.430571513891263]

output = shape.attr('worldSpace')
print(output.pointAtParam(0.5))
# Result: [-18.055226981265214, 0.0, 3.430571513891263]

# To get a plug output, pass plug=True in both cases:
print(shape.pointAtParam(0.5, plug=True))
# Result: NPCI.position
print(output.pointAtParam(0.5, plug=True))
# Result: NPCI.position

shape = r.PyNode('curve1').getShape()
print(shape.pointAtParam(0.5))
# Result: [-18.055226981265214, 0.0, 3.430571513891263]

output = shape.attr('worldSpace')
print(output.pointAtParam(0.5))
# Result: NPCI.position

Breaking Changes to NURBS and Bezier Curve Methods

Spun-Off Overloads

Further to the changes described above, some PyMEL methods on NURBS and Bezier curve shapes that were previously overloaded with features such as dynamic output (via a plug/p option) have been reverted, and the added functionality separated off. Specifically:

PyMEL Method

Paya Variant

closestPoint()

nearestPoint()

getCVs()

getControlVerts()

findParamFromLength()

paramAtLength()

findLengthFromParam()

lengthAtParam()

getPointAtParam()

pointAtParam()

getParamAtPoint()

paramAtPoint()

normal()

normalAtParam()

tangent()

tangentAtParam()

Background

This change was made because it was becoming very complicated to maintain ‘combination’ argument signatures on overloaded methods across shape and plug implementations.

Dropped Support for Full Components in Arguments

Geometry methods no longer accept full component arguments (for example instances of NurbsCurveParameter or string representations such as curve1.u[0.532]). Pass float or int instead:

param = curve.u[0.5] # yields NurbsCurveParameter instance
point = curve.pointAtParam(param) # fails with TypeError
point = curve.pointAtParam(0.5) # works

param = curve.u[0.5] # yields NurbsCurveParameter instance
point = curve.pointAtParam(param) # works
point = curve.pointAtParam(0.5) # also works

Background

This change was made to prevent expensive checks to disambiguate between string representations of components and attributes (where both could be used), and because such methods expect fully local information (e.g. the index of a CV on the current curve) anyway.

Other Breaking Changes on NURBS and Bezier Curves

  • distributeAimingMatrices() has been removed; use distributeMatrices(... chain=True) instead

  • fitChain() has been removed; use distributeJoints(... chain=True) instead

New Features

NURBS and Bezier Curves

Parallel Transport

Version 0.9 includes implementations for parallel transport inspired by a Houdini tutorial by Manuel Casasola Merkle, itself based on paper by A. Hanson and Hui Ma.

Parallel transport can be leveraged using a new workflow based on a up vector samplers. These are configured once per curve, and picked up implicitly by methods such as matrixAtParam() or distributeMatrices(). See Up Vector Sampling for more details.

General

Here’s a breakdown of all new methods and features on NURBS and Bezier curve shapes and plugs:

  • API / PyMEL methods that deal with curve lengths, such as pymel.core.nodetypes.NurbsCurve.length(), always run in local space. Paya counterparts, on the other hand, such as paramAtLength() or the overloaded length() are space-aware: When called on a shape, a worldSpace keyword argument is made available; when called on a plug, the space is determined by the choice of plug (for example, .local or .worldSpace on nurbsCurve nodes).

  • New methods have been added to configure up vector samplers: createUpVectorSampler() getUpVectorSamplers() getDefaultUpVectorSampler() clearUpVectorSamplers()

  • Compound sampler methods such as matrixAtParam() or distributeMatrices() will now use a default up vector sampler if one has been configured. New upVectorSampler/ups and defaultToNormal/dtn options have been added to override this default behaviour.

  • Discrete up-vector sampling operations are also available for lower-level hacking: upVectorAtParam() upVectorAtPoint() upVectorAtFraction() upVectorAtLength()

  • Anchor-aware sampler methods on BezierCurve now include: paramAtAnchor() paramsAtAnchors() getCVAtAnchor() getCVsAtAnchor() pointAtAnchor() getControlVerts() matrixAtAnchor()

  • Anchor-aware editor methods on BezierCurve now include: inbetweenAnchors() subdivideAnchors()

  • A new insertKnot() method has been added to NurbsCurve.

  • A dispCV/dcv=True option has been added to createShape() on paya.runtime.plugs.NurbsCurve and to create() on NurbsCurve.

Miscellaneous Additions

  • A pn shorthand has been added for pymel.core.general.PyNode on paya.runtime, allowing you to write loc = r.pn('locator1')` instead of loc = r.PyNode('locator1').

Bugs Fixed

  • Editor methods on NurbsCurve and BezierCurve create intermediate shapes when the type changes between NURBS and Bezier

  • paya.runtime.data.Vector.angle() returns float instead of Angle

  • paya.runtime.plugs.Vector.get() sometimes returns tuple instead of Vector

  • paya.runtime.nodes.BezierCurve.create() creates unexpected shapes because the generated knot list is not suitable for a Bezier curve

  • paya.runtime.plugs.Math1D.acos() leads to excessive error warnings when the input range is outside 0.0 -> 1.0
